# Retool for Startups — Application Archetype

## What this program wants

- **Form shape:** standard company basics plus one differentiating free-text field: **"Anything else we should know?"**
- **Tone:** warm, specific, founder-voiced. This field is read by humans who like builders.
- **What wins:** naming the *internal operations* you'll run on Retool (ops dashboards, pipeline monitoring, review queues) — not customer-facing UI. Retool wants operators, and it doubles as a referral hub: acceptance can unlock referrals into other vendor programs.

## Templated answer

**"Anything else we should know?"**

```
Genuinely stoked about this. I'm building {{company_name}} — {{one_liner}} —
and Retool is exactly the kind of tool I've been wishing existed.

Short version: I'm not trying to build a fancy admin UI for every part of the
product. I need fast, reliable internal tooling so I can actually operate
what I'm shipping. [Name 2–3 real operator needs, e.g.: "our data pipelines
need an operator view — ingestion health, review queues, entity QA";
"I want one place to monitor background jobs and dispatch work without
writing another bespoke panel"; "cross-team ops dashboards".]

Retool is the move because I can wire it straight to our [database/API stack,
e.g. Postgres backends and internal APIs], ship useful tools in hours instead
of weeks, and stay focused on the product. That speed is crucial for a team
our size ({{headcount}}).

Really grateful you're doing the startups program. Happy to share what we
build, give feedback, or be a reference if it's useful. Thanks for reading!
```

## Notes from a successful application

- Anchor every claim in a named internal-ops need; "we'll build admin panels"
  is too generic to win.
- The "hours instead of weeks" velocity frame is the value Retool sells —
  reflect it back with your own specifics.
- After acceptance, ask about partner referrals — this program can open doors
  at other vendors.
